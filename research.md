## 2. ЧТО ЗАБЫВАЮТ НА B2B-ЛЕНДИНГАХ (критично для нас)

1. **Кто ответит на заявку** — написать: "Ответит инженер с опытом X лет, не менеджер по продажам"
2. **Срок изготовления прямо в результатах** — "Стандарт: 7–10 дней, срочно: 3–5 дней"
3. **Пример неправильного расчёта** — "Ошибка в 10% по давлению = цилиндр не тянет нагрузку"
4. **NDA для чертежей** — многие клиенты не высылают чертежи без NDA
5. **Документация** — паспорт изделия, акт испытаний, сертификат материалов — упомянуть явно
6. **"Не знаю параметры"** — дать опцию: "Опишите задачу словами → инженер перезвонит"

## 3. UX КАЛЬКУЛЯТОРА — ЧТО УЛУЧШИТЬ

### Добавить немедленно:
- **Прогресс-бар** "Шаг 1 из 2 — Определяем диаметр"
- **Сводка этапа 1 на этапе 2** — показать компактно: "Диаметр гильзы: 63 мм (из шага 1)"
- **Кнопка "Назад"** — не заставлять пользователя перезаполнять
- **Inline-подсказки** для полей:
  - Давление: "Типовые значения: 10–16 МПа (мобильная техника), 20–25 МПа (промышленность)"
  - Время цикла: "Время полного хода штока, влияет на расход насоса"

### Схема гидроцилиндра (SVG):
```
Разместить справа от полей (десктоп) или выше (мобиль)
Показывать: D (гильза), d (шток), L (ход)
При вводе значений — обновлять надписи на схеме в реальном времени
```

### Блок результата:
```
┌─────────────────────────────────┐
│ ✓ Расчёт завершён               │
│                                 │
│ Диаметр гильзы:  63 мм  ← большой, оранжевый
│ Диаметр штока:   40 мм           │
│ Объём:           1.24 л          │
│ Расход насоса:   14.9 л/мин      │
│ Артикул:  ГЦ-63/40-300          │
│                                 │
│ [Скопировать]  [Распечатать]    │
│ [Отправить заявку →]            │
└─────────────────────────────────┘
```

### Текст для копирования (в буфер):
```

Усилие: 50 000 Н | Давление: 16 МПа | Ход: 300 мм
```

## 4. TRUST SIGNALS — ЧТО ПОКАЗЫВАТЬ

### В hero-секции (4 цифры):
- Лет на рынке: `__` лет
- Реализованных проектов: `__`+
- Регионов доставки: `__`+
- Гарантия: `__` года

### Блок сертификатов:
- ГОСТ 3822-79 (гидроцилиндры)
- ISO 9001 (если есть)
- ТР ТС 010/2011
- Испытание каждого изделия при 150% рабочего давления

### Структура кейса:
```
Отрасль | Компания (ООО..., город)
Задача: [1–2 предложения с болью клиента]
Решение: [что именно сделали, параметры]
Результат: [конкретные цифры — время, деньги, срок работы]
Фото: [процесс или готовое изделие]
```

## 5. ФОРМА ЗАПРОСА КП — ОПТИМАЛЬНАЯ СТРУКТУРА

**Только 5 полей** (максимум 3 минуты):
1. Имя + должность
2. Компания + отрасль (список)
3. Что нужно (свободный текст, 2–3 слова)
4. Телефон (с маской +7)
5. Когда нужно (три варианта: срочно / месяц / планирую)

**Рядом с кнопкой**: "Или позвоните: +7 (3412) 77-57-04"

**После отправки**:
> ✓ Заявка принята. Инженер перезвонит в рабочее время (пн–пт 8:30–17:00).
> Если заявка в выходной — ответим в понедельник.

**Опция для неуверенных**: "Не знаете параметры? Опишите задачу словами — инженер поможет с расчётом"

## ЭКСПЕРТНАЯ ОЦЕНКА

> Три независимые точки зрения на проект лендинг-калькулятора подбора гидроцилиндров

---

### 🎨 ЭКСПЕРТ 1: UX-дизайнер (10 лет опыта)

#### Типичные ошибки новичков в структуре

| Ошибка | Почему плохо | Как исправить |
|--------|-------------|---------------|
| Форма «для всего» — и расчёт, и заявка в одном блоке | Пользователь не понимает, что он сейчас делает | Чёткое разделение: сначала калькулятор, потом форма заявки |
| Кнопка «Отправить» без объяснения, что будет дальше | Тревога: куда уйдут данные, кто перезвонит | Писать конкретно: «Инженер перезвонит в течение 2 часов» |
| Результат калькулятора исчезает при переходе к форме | Пользователь теряет контекст — зачем он вообще здесь | Показывать результат расчёта рядом с формой заявки |
| Нет состояния «в процессе» — поля заполнены, но ничего не произошло | Ощущение сломанного интерфейса | Мгновенный пересчёт при вводе (уже реализовано ✓) |
| Мобиль — afterthought, а не основа | 30–40% B2B-трафика с телефона (инженер в цехе) | Проектировать mobile-first, проверять на 375px |

#### Что ОБЯЗАТЕЛЬНО должно быть

1. **Прогресс-бар** «Шаг 1 из 2» — инженер должен понимать, где он находится и сколько осталось
2. **Сводка этапа 1 видна на этапе 2** — «Диаметр гильзы: 80 мм (из шага 1)» компактной плашкой
3. **Кнопка «Назад»** между этапами — без неё пользователь перезаполняет всё с нуля
4. **Пустое состояние** — до первого ввода показывать пример заполненного расчёта или подсказку
5. **Явный CTA после результата** — не просто артикул, а «Получить КП на этот цилиндр»

#### Что раздражает и заставляет уйти

- Поле без подсказки о единицах измерения или диапазоне — «а что вводить, Н или кН?»
- Слишком много полей сразу — эффект «анкеты на 3 страницы»
- Мелкий шрифт на мобильном, тапабельные элементы меньше 44px
- Нет обратной связи при ошибке — поле просто не считает
- Долгий скролл до результата после ввода данных

### 📊 ЭКСПЕРТ 2: Маркетолог по конверсии

#### Оффер, который зацепит

**Слабый оффер** (как у многих): «Гидроцилиндры на заказ»
**Сильный оффер**: «Подберите гидроцилиндр за 3 минуты — получите КП с ценой и сроком»

Формула: **результат** + **время/усилие** + **конкретное следующее действие**

Примеры для hero:
- «Введите усилие и давление — калькулятор выдаст артикул, объём масла и подачу насоса»
- «Без звонков и писем — рассчитайте сами, отправьте заявку одной кнопкой»

#### Триггеры доверия — что и где разместить

| Триггер | Где разместить | Почему работает |
|---------|---------------|-----------------|
| Лет на рынке + число проектов | Hero, под заголовком | «Они делали это раньше» |
| «Ответит инженер, не менеджер» | Рядом с формой заявки | Снимает страх некомпетентного ответа |
| Испытание при 150% давления | Блок сертификатов | Говорит о качестве на языке инженера |
| NDA до обсуждения чертежей | FAQ или рядом с формой | Снимает главный барьер у производства |
| Конкретные сроки: 7–10 дней стандарт, 3–5 срочно | В результатах расчёта | Клиент планирует — ему нужны цифры |
| Реальные кейсы с параметрами | Отдельная секция | «Делали ли они похожее?» — главный вопрос B2B |

#### Путь посетителя (воронка)

```
Зашёл → Hero (понял, что здесь) → Калькулятор (вовлёкся)
→ Получил результат (ценность получена) → Увидел КП-кнопку
→ Убедился через кейсы/сертификаты → Оставил заявку
```

**Ключевой момент**: ценность (результат расчёта) должна быть получена ДО того, как просить контакт. Калькулятор — это lead magnet.

**Micro-конверсии** на пути:
1. Пользователь ввёл данные в этап 1 — уже вовлечён
2. Нажал «Скопировать результат» — высокое намерение
3. Нажал «Печать» — скорее всего покажет коллегам

### 👤 ЭКСПЕРТ 3: Потенциальный клиент — владелец бизнеса / главный инженер

#### Первые 5 секунд — что он видит

**Смотрит на:**
1. Заголовок — «это про мою задачу?»
2. Есть ли калькулятор / инструмент — «могу ли я сделать что-то прямо сейчас?»
3. Телефон и название компании — «кто это вообще?»
4. Визуальную сложность — «разберусь ли я быстро?»

**Думает:**
- «Мне нужен гидроцилиндр под конкретный пресс — здесь помогут?»
- «Они производят сами или перепродают?»
- «Сколько это будет стоить?»

#### Что заставит остаться

- Калькулятор работает сразу, без регистрации и звонка
- Результат содержит артикул — значит, они знают своё дело
- Видно, что компания реальная: телефон, адрес, лет на рынке
- Есть похожие кейсы («делали для металлургии» / «делали для прессов»)
- Сроки и цена — хотя бы диапазон — указаны явно

#### Что заставит закрыть вкладку

- Нет цены совсем, даже диапазона — значит, «дорого»
- Форма заявки появляется раньше, чем получена ценность
- Непонятно, кто будет отвечать и когда
- Нет ничего про гарантию и документацию
- Сайт выглядит как сделанный «лишь бы был» — нет доверия к качеству продукта

## 8. TELEGRAM MINI APP — КАНАЛ ДЛЯ КАЛЬКУЛЯТОРА

### Зачем это нужно

Telegram Mini App (TMA) — это веб-приложение, встроенное в Telegram. Для Промгидравлики это не замена сайту, а дополнительный канал: инженер в цехе открывает бота, вводит параметры, получает результат и сразу отправляет его менеджеру в чат. Никакого перехода в браузер, никакой регистрации.

**Ключевой аргумент:** калькулятор уже работает на чистом HTML/CSS/JS — его можно встроить в TMA практически без переработки.

---

### Что реально сделать в TMA

| Функция | Сложность | Польза |
|---------|-----------|--------|
| Калькулятор подбора гидроцилиндра | Низкая (перенос существующего) | Основной инструмент |
| Отправка результата в чат с менеджером | Низкая (`Telegram.WebApp.sendData`) | Замена форме заявки |
| Сохранение истории расчётов | Низкая (localStorage или `CloudStorage` API) | Инженер возвращается к прошлому расчёту |
| Уведомление менеджера о новой заявке | Низкая (через бот-API) | Заявки сразу в Telegram — не теряются |
| Каталог стандартных артикулов | Средняя | Быстрый поиск без расчёта |

---

### Технические детали (для чистого HTML/JS)

**Инициализация:**
```html
<script src="https://telegram.org/js/telegram-web-app.js"></script>
<script>
  const tg = window.Telegram.WebApp;
  tg.ready();
  tg.expand(); // развернуть на весь экран
</script>
```

**Отправка данных в бот:**
```js
// Отправляет JSON в обработчик бота (событие web_app_data)
tg.sendData(JSON.stringify({
  force: 50000,
  pressure: 16,
  stroke: 300,
  diameter: 63,
  rod: 40,
  article: 'ГЦ-63/40-300'
}));
```

**MainButton (кнопка внизу экрана):**
```js
tg.MainButton.setText('Отправить заявку');
tg.MainButton.show();
tg.MainButton.onClick(() => tg.sendData(...));
```

**Тема — брать из Telegram, не задавать вручную:**
```js
// Переменные цвета доступны через CSS var(--tg-theme-bg-color) и т.д.
// Для тёмной темы Telegram наш тёмный фон (#060810) хорошо подходит
document.body.style.background = tg.themeParams.bg_color || '#060810';
```

**CloudStorage (если нужна история между сессиями):**
```js
tg.CloudStorage.setItem('last_calc', JSON.stringify(result));
tg.CloudStorage.getItem('last_calc', (err, val) => { ... });
```

### UX в среде Telegram — ограничения и решения

| Ограничение TMA | Решение |
|----------------|---------|
| Нет URL-навигации (нет кнопки «назад» браузера) | Реализовать BackButton: `tg.BackButton.show()` и обрабатывать событие `backButtonClicked` |
| Высота экрана меняется при открытии клавиатуры | Слушать `tg.onEvent('viewportChanged', ...)` и пересчитывать layout |
| Нет `alert()` / `confirm()` привычных в браузере | Использовать `tg.showAlert()` и `tg.showConfirm()` из Telegram API |
| Тапабельные элементы должны быть ≥44px | Уже требуется по CLAUDE.md — соответствие есть |
| Шрифты системные (не Arial Black в мобиле) | Задать явно `font-family: 'Arial Black', Arial, sans-serif` в CSS |

## ЭКСПЕРТНАЯ ОЦЕНКА: КАТАЛОГ В TELEGRAM MINI APP

> Три независимые точки зрения на идею каталога подбора гидроцилиндров в формате Telegram Mini App

---

### 🎨 UX-ДИЗАЙНЕР — каталог в TMA

#### Типичные ошибки новичков

| Ошибка | Почему плохо | Как исправить |
|--------|-------------|---------------|
| Забывают про BackButton | Пользователь не может вернуться — закрывает приложение | Всегда показывать `tg.BackButton` при переходе вглубь каталога |
| Фильтры как всплывающее окно поверх списка | В TMA нет нормального z-index контекста, перекрывает всё | Фильтры — отдельный экран, а не модалка |
| Длинный список без группировки | Скролл до конца — пользователь теряется | Группировать по диаметру гильзы или типу: ГЦ-50, ГЦ-63, ГЦ-80... |
| Карточка товара с 15 параметрами | Инженер не может быстро найти нужное | В карточке — 3–4 ключевых параметра, остальное — по тапу |
| Клавиатура перекрывает поисковое поле | Поле уходит под клавиатуру, не видно что вводишь | Слушать `viewportChanged`, прокручивать к активному полю |
| Нет состояния «Нет результатов» | Пустой экран выглядит как баг | Показывать: «Не нашли — опишите задачу, подберём» + кнопка |
| Тапабельные элементы меньше 44px | На мобиле промахиваются | Карточки каталога — минимум 64px высотой |

#### Что ОБЯЗАТЕЛЬНО должно быть

1. **Поиск + фильтр** — инженер знает диаметр или усилие, ему нужна быстрая фильтрация
2. **Карточка каталога: 3–4 параметра** — D (гильза), d (шток), L (ход), P (давление)
3. **Экран товара** — полные параметры + кнопка «Добавить в заявку»
4. **Корзина заявки** — накапливать позиции, отправлять одним запросом
5. **MainButton** с числом позиций в заявке — всегда видна, не мешает
6. **BackButton** — на каждом экране кроме главного
7. **Haptic feedback** — при добавлении в заявку (`tg.HapticFeedback.notificationOccurred('success')`)
8. **Пустое состояние** при первом открытии — подсказка как искать

#### Что бесит пользователей

- Нет поиска — приходится листать весь каталог
- Результаты поиска появляются с задержкой (дольше 300 мс)
- При возврате из карточки — список прокручен в начало (потерял место)
- Нет подтверждения, что позиция добавлена в заявку
- Мелкие параметры — инженер в цехе не может прочитать
- Отправил заявку — тишина, непонятно дошло ли

---

### 📊 МАРКЕТОЛОГ — каталог в TMA

#### Что заставит оставить заявку

**Главный принцип B2B в Telegram:** никаких регистраций и форм. Инженер выбрал — нажал — менеджер увидел. Всё.

| Элемент | Почему работает |
|---------|----------------|
| Заявка = имя Telegram + позиции | Не нужно ничего вводить — контакт уже есть |
| Цена «от X ₽» в карточке | Убирает страх «дорого» без звонка |
| Значок «В наличии» / «Под заказ 7–10 дней» | Инженер сразу планирует |
| Кнопка «Поделиться позицией» | Инженер показывает коллеге прямо в Telegram |
| «Популярные» / «Часто заказывают» | Снижает тревогу выбора — «другие берут это» |
| Быстрый ответ от менеджера (< 2 часов) | Обещать и выполнять — повторные заявки |

#### Структура каталога, которая конвертирует

```
[Главный экран]
  Поиск  +  Популярные категории (6 плашек)
  Недавно смотрели (если есть история)

[Список по категории]
  Карточка: D/d/L + цена от + наличие
  Фильтр: давление / ход / наличие

[Экран позиции]
  Параметры полностью
  Схема (SVG)
  Цена диапазон
  "Добавить в заявку" (MainButton)

[Экран заявки]
  Список позиций
  Поле "Комментарий" (опционально)
  "Отправить менеджеру" (MainButton)
```

#### Micro-конверсии, которые важны

1. Пользователь открыл каталог → уже тёплый лид (бот знает username)
2. Просмотрел 3+ позиции → высокий интерес → можно отправить сообщение в чат
3. Добавил в заявку → почти конверсия → не терять этот момент
4. Поделился позицией → органический охват → другой инженер = новый лид

#### Оффер в описании бота (первые 160 символов)

> «Каталог гидроцилиндров: подберите по диаметру и давлению. Заявка — в один тап, менеджер ответит за 2 часа. Промгидравлика — от 1 шт.»

### 👨‍💻 TELEGRAM-РАЗРАБОТЧИК — ограничения и подводные камни

#### Что работает хорошо

| Возможность | Как использовать |
|-------------|-----------------|
| `MainButton` | Кнопка «Отправить заявку» всегда внизу — не нужен кастомный footer |
| `BackButton` | Нативная навигация без роутера — один стек экранов |
| `HapticFeedback` | Подтверждать добавление в заявку тактильно |
| `CloudStorage` | Хранить корзину заявки между сессиями (до 1024 ключей, 4096 байт каждый) |
| `themeParams` | Цвета Telegram — тёмная тема уже подходит под стиль Промгидравлики |
| `openLink` | Открыть страницу p-gidravlika.ru без выхода из бота |
| `showConfirm` | Подтверждение отправки заявки — нативный диалог |
| `initDataUnsafe.user` | Username + имя пользователя — контакт без ввода |

#### Ограничения — что лучше не делать

| Нельзя / не стоит | Почему | Альтернатива |
|-------------------|--------|-------------|
| `alert()` / `confirm()` | Не работает в TMA | `tg.showAlert()` / `tg.showConfirm()` |
| Загрузка файлов (чертежи) | TMA не поддерживает `<input type="file">` нативно | Просить прислать чертёж отдельным сообщением боту |
| Сложный роутер с историей | BackButton управляет только одним уровнем | Плоская структура: главный → список → карточка → заявка |
| LocalStorage для критичных данных | Очищается при обновлении TMA | CloudStorage API для корзины и истории |
| Фиксированная высота экрана | `window.innerHeight` меняется при открытии клавиатуры | `tg.viewportHeight` и слушать `viewportChanged` |
| CSS `position: fixed` для шапки | Конфликтует с нативным header TMA | `position: sticky` или отступы через переменные |
| CORS-запросы к стороннему API | Браузер блокирует | Все запросы — через свой backend/serverless |
| Хранить токен бота в JS | Утечёт из клиентского кода | Только на сервере, в переменных окружения |

#### Архитектура MVP (минимально рабочий вариант)

```
[Vercel]
  /tma/catalog.html  ← TMA (чистый HTML/JS)
  /api/products.js   ← Список позиций из JSON или БД
  /api/order.js      ← Принять web_app_data → отправить боту → уведомить менеджера

[Telegram Bot]
  /start → кнопка "Открыть каталог" (WebApp button)
  web_app_data → пересылка заявки в чат менеджера

[Данные]
  products.json — статика, каталог позиций (можно начать без БД)
  CloudStorage — корзина пользователя
```

#### Критичный код для инициализации

```js
const tg = window.Telegram.WebApp;
tg.ready();
tg.expand();

// Подстраиваться под высоту экрана
tg.onEvent('viewportChanged', () => {
  document.documentElement.style.setProperty(
    '--tg-viewport-height', tg.viewportHeight + 'px'
  );
});

// Получить контакт пользователя без форм
const user = tg.initDataUnsafe?.user;
// user.username, user.first_name — готово к отправке с заявкой
```
